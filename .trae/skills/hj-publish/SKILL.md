# hj-publish：灰金重组公众号全流程发布引擎

按顺序执行以下子skill，每步执行完将关键结论给用户确认，确认后再进下一步。

---

## 核心原则

1. **按步执行** — 每步调用对应的子skill，输出结果给用户确认，确认后再进下一步
2. **流程不可跳过** — 固定10步，不可交换顺序
3. **问题当步闭环** — 每一步发现的明确问题，原则上必须在本步给出处理方案、形成修订稿或待用户确认的替换句；不得只说“后面再改”后继续推进
4. **第6步总回查** — 首尾段优化前，必须先回查②内容诊断、③AI检测、⑤法律审查问题是否已在对应步骤处理；若发现漏处理，必须回到对应步骤补闭环，再改首尾段
5. **品牌推广区由 hj-wechat-html 一体输出** — 灰金风格（brand-imprint）已包含完整的品牌推广区 + 免责声明 + 版权信息，不需要手动拼接
6. **输出位置** — 最终 HTML 放 `$PROJECT_ROOT/content-system/content/output/`
7. **品牌封面必须确定性合成** — 图片模型只生成无文字、无 Logo 底图；标题和正式转曲 Logo 必须后期直接合成，禁止模型重绘品牌内容
8. **品牌规范优先** — 进入第⑧步前，必须完整读取 `$PROJECT_ROOT/docs/灰金品牌视觉素材使用规范.md`
9. **官网同步不等于生产上线** — 第⑩步只更新正式官网源代码并完成本地验证；未经用户再次明确确认，不得执行 Git 推送或 Vercel 生产部署

## 全流程

```
选题 / Markdown 稿
    ↓
① hj-topic       → 选题归类 + 栏目判断 + 转化入口
    ↓ (用户确认)
② hj-content     → 五维诊断
    ↓ (用户确认)
③ hj-ai-check    → AI 特征扫描 + 发布流程内逐条改写闭环
    ↓ (用户确认)
④ hj-title       → 标题推荐 + 选定
    ↓ (用户确认)
⑤ hj-legal-check → 法律合规审查
    ↓ (用户确认)
⑥ 首尾段优化     → 前序问题回查 + 手动改写首段 + 末段
    ↓ (用户确认)
⑦ hj-wechat-html → 风格: brand-imprint → 输出完整 HTML（正文+品牌推广区）
    ↓ (用户确认)
⑧ 封面配图与确认 → AI 无字底图 + 确定性标题/Logo 合成 + 输出校验
    ↓ (用户确认)
⑨ hj-ingest 入库 → 源文件归档 + 统一归类 + 拆解卡确认 + 索引登记 + 验证
    ↓ (用户确认)
⑩ 同步灰金官网    → 单篇预检 + 专栏数据/正文/配图更新 + 本地验证
    ↓
人发布到公众号
```

### ① hj-topic 选题立项

**输出**：选题归类、目标读者、困境阶段、核心误判、推荐写法、转化入口和延伸选题。

如果用户已经提供完整 Markdown，也必须先用 `hj-topic` 做一次轻量立项校准，确认这篇文章属于哪个栏目、服务哪个困境阶段、最终导向哪个转化入口。不要在这一步写入任何文件。

### ③ hj-ai-check AI 特征扫描与改写闭环

**输出**：AI 特征检测报告、逐条处理表、可替换句或修订方向、处理后待确认稿。

`hj-ai-check` 子skill 的职责是识别 AI 写作特征，默认只检测不改写。但在 `hj-publish` 发布流程中，不能停在检测报告。主流程必须承接检测结果，完成以下闭环：

1. 按 `hj-ai-check` 报告逐条列出问题；
2. 从第 1 处开始逐条处理，每次只处理 1 处问题；
3. 每一处先说明问题，再给 1-2 个候选改法或处理方向；
4. 等用户明确确认、选择或补充后，记录该处最终处理结果；
5. 当前问题未确认前，不得进入下一处问题；
6. 对每一条给出处理方式：删除、弱化、补案例、补前提、改表达、保留并说明理由；
7. 对可直接处理的句子，给出可替换句或局部修订稿；
8. 对需要用户补充真实经历、数据、案例来源的问题，标记为「待用户补充」，不得假造细节；
9. 全部问题逐条确认后，输出「AI 检测问题处理清单」，明确每一条状态：已处理 / 保留 / 待用户补充；
10. 用户确认第③步整体结果后，才能进入标题步骤。

第③步必须完成 AI 问题的逐条处理和用户确认，不得一次性给出全部改写稿让用户批量确认，也不得把「AI 味后面一起改」作为进入第④步的理由。第⑥步只负责回查第③步是否漏改，不替代第③步改写闭环。

如果 AI 检测发现的问题同时属于法律风险或事实核验风险，应在本步先做表达弱化，并把该问题带入第⑤法律审查继续核验。

### ⑤ hj-legal-check 法律审查与合规改写闭环

**输出**：法律合规审查报告、逐条风险处理表、可替换句或删改意见、处理后待确认稿。

`hj-legal-check` 子skill 的职责是检测法律合规风险，默认不直接修改原文文件。但在 `hj-publish` 发布流程中，不能停在审查报告，也不能把风险点顺延到第⑥步。主流程必须承接审查结果，完成以下闭环：

1. 先按 `hj-legal-check` 输出法律依据核验清单和风险点；
2. 从第 1 处风险开始逐条处理，每次只处理 1 处；
3. 每一处先说明法律/合规风险，再给 1-2 个候选处理方式：删除、弱化、补免责声明、补条件、改成经验表述、改成需核验表述；
4. 等用户明确确认、选择或补充后，记录该处最终处理结果；
5. 当前风险未确认前，不得进入下一处风险；
6. 对需要官方公告、法院文书、监管文件或专业人士复核的问题，标记为「待人工核验」，不得编造事实或直接下结论；
7. 全部风险逐条确认后，输出「法律风险处理清单」，明确每一条状态：已处理 / 删除 / 保留并说明理由 / 待人工核验；
8. 用户确认第⑤步整体结果后，才能进入第⑥步首尾段优化。

第⑤步必须完成法律风险的逐条处理和用户确认，不得把「法律风险第⑥步一起改」作为进入第⑥步的理由。第⑥步只负责回查第⑤步是否漏改，不替代第⑤步合规改写闭环。

### ⑥ 首尾段优化与前序问题回查

**输出**：前序问题回查表、处理后的正文关键风险句、优化后的首段和末段。

进入首尾段优化前，必须先检查前面步骤留下的问题是否已经处理：

1. 回查② `hj-content` 的结构、案例、专业深度问题；
2. 回查③ `hj-ai-check` 的 AI 指纹和逐条改写清单；
3. 回查⑤ `hj-legal-check` 的事实核验、绝对化承诺、免责声明和服务边界风险是否已在第⑤步逐条处理；
4. 对未处理项先给出修订句或明确删改意见；
5. 确认前序问题已处理或已标记为待用户补充后，再优化首段和末段。

第⑥步不是用来替代前面步骤的改写闭环，而是发布前的总回查。若发现前面步骤漏改，必须先补漏，再进入首尾段优化。

### ⑧ 品牌封面配图与确认输出

**输出**：公众号无字底图、添加准确标题和正式 Logo 的发布候选图，以及 HTML 与图片输出检查结果。

第⑧步开始前，必须完整读取：

```text
$PROJECT_ROOT/docs/灰金品牌视觉素材使用规范.md
```

并按以下顺序执行：

#### ⑧-1 确认配图输入

从前序已确认结果中提取：

- 第④步确认的最终文章标题；
- 根据品牌规范确定封面标题：正式标题不超过 24 个中文字符且排版清晰时可直接使用；过长或过密时，提出 8—16 个中文字符的短封面标题并单独取得用户确认；
- 第⑥步确认的文章核心判断和画面情绪；
- 用户指定的画面比例；用户未指定时默认 2.35:1；
- 品牌规范登记的主色 `#AB1942`；
- 品牌规范登记的正式转曲 Logo 路径和 SHA-256。

标题、文章核心判断或 Logo 真源未确认时，不得生成发布候选图。

#### ⑧-2 生成无文字、无 Logo 底图

如果当前环境提供 `imagegen` 或其他图片生成 Skill，必须先读取其完整说明并按其流程生成底图。提示词必须明确：

- 不生成 Logo、品牌名称和口号；
- 不生成文章标题；
- 不生成任何可读中英文、数字标签或伪文字；
- 为 Logo 和标题预留干净区域；
- 以 `#AB1942` 为品牌主色；
- 画面积极、克制、协调，与文章核心判断一致；
- 避免债务恐吓、钞票、锁链、破碎玻璃、红色警报、法槌和通用握手图。

生成后必须保存独立底图，不得把模型生成的 Logo 或文字修修补补后继续使用。底图含伪文字、伪品牌或无意义数字时，应重新生成或清理为无字底图。

#### ⑧-3 确定性合成标题和正式 Logo

图片模型不得参与本步骤。

1. 从最终标题原文直接复制标题，不手工重新输入；
2. 如果使用用户确认的短封面标题，只修改图片文字，不修改公众号后台正式标题；
3. 默认只显示一个封面标题和正式 Logo，不添加副标题、摘要、作者、日期、栏目名、编号、行动号召、网址或额外口号；
4. 标题原则上不超过两行，每行宜控制在 12 个中文字符以内，最多突出一个关键词或短语；
5. 不得为了塞入长标题而缩小到手机端难以辨认，文字区域原则上不超过画面宽度的 45%；
6. 优先使用本 Skill 已验证的脚本添加标题和 Logo：`$PROJECT_ROOT/.trae/skills/hj-publish/scripts/compose_wechat_cover.py`；
7. 脚本只读取 `$PROJECT_ROOT/content-system/assets/brand/manifest.json` 登记且哈希一致的正式 Logo 受控派生件；
8. 浅色背景使用标准色转曲版，深色背景使用透明白色转曲版；
9. Logo 只允许等比例缩放和平移，不得改字、裁切、拉伸、旋转、改色或添加效果；
10. 无法直接叠加正式 Logo 时，只交付无字底图并标记「待品牌合成」，不得用模型近似 Logo 代替；
11. 标题和 Logo 未通过校验时，不得标记为发布候选图。

运行示例：

```bash
python3 "$PROJECT_ROOT/.trae/skills/hj-publish/scripts/compose_wechat_cover.py" \
  --base "$PROJECT_ROOT/content-system/content/output/{文章短标题}_公众号配图_底图_v01.png" \
  --output "$PROJECT_ROOT/content-system/content/output/{文章短标题}_公众号配图_发布候选_v01.png" \
  --article-title "{公众号后台正式标题}" \
  --cover-title "{第一行}|{第二行}" \
  --accent "{最多一个强调短语}" \
  --logo-variant color \
  --logo-position top-left \
  --title-side right
```

要求：

- 使用包含 Pillow 的 Python 运行环境；Codex 桌面环境应先读取工作区依赖路径；
- 不得自动安装依赖到项目或修改项目运行环境；
- 浅色背景使用 `--logo-variant color`，深色背景使用 `white`；
- 不需要强调短语时省略 `--accent`；
- 输出文件已存在时脚本默认停止，只有用户明确要求覆盖时才可使用 `--overwrite`；
- 脚本成功后必须读取同名 `.report.json`，不得只看命令退出码。

#### ⑧-4 文件命名与保存

至少保存：

```text
$PROJECT_ROOT/content-system/content/output/{文章短标题}_公众号配图_底图_v01.png
$PROJECT_ROOT/content-system/content/output/{文章短标题}_公众号配图_发布候选_v01.png
$PROJECT_ROOT/content-system/content/output/{文章短标题}_公众号配图_发布候选_v01.report.json
```

规则：

- 底图和发布候选图不得使用同一个文件名；
- 未经用户确认不得覆盖旧版本；
- 用户确认前不得使用“最终”“定稿”等状态词；
- 同时确认 HTML 文件仍在 `$PROJECT_ROOT/content-system/content/output/` 中。

#### ⑧-5 校验与用户确认

向用户报告并逐项确认：

- 图片实际尺寸和比例；
- 文章标题逐字核对结果；
- 封面是否只保留一个标题，长标题是否已使用用户确认的短封面标题；
- Logo 使用的真源文件和 SHA-256；
- Logo 是否完整、等比例、未改色；
- 底图是否无伪文字、伪 Logo 和无意义数字；
- 手机缩略图中的标题可读性；
- HTML 和两张图片的保存路径。
- `.report.json` 的保存路径、脚本版本、底图哈希、输出哈希和字体哈希。

发现任何错字、Logo 小字变化、模型重绘品牌标志、比例错误或明显主题偏离时，必须在第⑧步退回处理。用户明确确认发布候选图和 HTML 后，才能进入第⑨步。

### ⑨ hj-ingest 入库（内容资产归档）

**输出**：文章源文件 + 统一拆解卡 + 索引登记 + QST/CON/OPI/CAS/SOL 内容单元 + 金句拆解 + 处理状态

确认 HTML 输出无误后，调用 `hj-ingest`，将文章转化为可复用的内容资产。第⑨步仍有一次独立的“拆解卡确认”，不能因为前面已经确认正文和 HTML 就跳过：

```

① 源文件归档
   ├ 确认文章 MD 源文件在 $PROJECT_ROOT/content-system/content/articles/ 中
   ├ 如发布时是从草稿直接生成 HTML 的，需将标准投稿 Markdown 写入 articles/
   └ 文件命名：以文章核心内容命名，如《干了10年困境重组》

② 索引登记
   ├ 打开 $PROJECT_ROOT/content-system/content/index.json
   ├ 在 articles 数组中新增条目（id、title、date、type、tags、word_count、file）
   └ 更新 version 和 updated 字段

③ 统一归类
   ├ 继承第①步已确认的栏目、目标读者、困境阶段和转化入口
   ├ 补齐主话题、读者问题、核心误判、推荐写法和叙事入口
   └ 栏目只能使用：困境故事 / 老板避坑 / 重组判断课 / 行业冷眼

④ 元素拆解与视角回查
   ├ 通读全文，识别 QST 问题、CON 概念、OPI 观点、CAS 案例、SOL 方案和金句
   ├ 至少包含 1 个 QST 和 1 个 OPI；CON/CAS/SOL 按正文实际情况提取
   ├ 每篇文章优先拆 3-5 个高复用概念，不为凑数制造伪概念
   ├ 检查读者问题、可执行获得感、作者经验开头、品牌自述和重复稿候选
   ├ 按 processing/decomposition-schema.json 生成拆解卡草案
   └ 把归类、内容单元、金句和视角回查结果给用户确认

⑤ 拆解卡与概念写入
   ├ 用户确认后写入 processing/articles/{文章ID}.json
   ├ 每个概念写一个 .md 文件到 concepts/，格式：
     ---
     title: "概念名称"
     type: 概念/方案/法律概念
     source_articles: [文章ID]
     related_concepts: [关联概念ID]
     tags: [标签]
     ---

     # 概念名称

     ## 定义

     ## 在债务/重组中的表现

     ## 来源
   └ 每个概念需满足：有清晰定义、可跨文章复用、与现有概念不重复

⑥ 索引更新
   ├ 在 index.json 的 concepts 数组中新增条目
   ├ 关联 related_articles 到对应文章 ID
   └ 更新 version 和 updated 字段

⑦ 金句抽取
   ├ 从文章中提取 3-10 条可独立传播的金句
   ├ 写入 $PROJECT_ROOT/content-system/content/quotes/{文章ID}金句.md
   └ 在 index.json 的 quotes 数组中新增条目

⑧ 状态与验证
   ├ 更新 processing/article-status.json
   ├ 拆解卡必须符合 decomposition-schema.json
   ├ json格式校验：index.json 必须可解析
   ├ 文件一致性：所有 concepts/ 文件都存在
   └ 引用关系：所有 related_articles 指向已存在的文章 ID
```

### ⑩ 同步灰金官网（本地正式源）

**输出**：官网专栏正文、专栏数据、必要配图、同步检查结果和本地预览地址。

第⑨步完成且用户确认入库结果后，必须询问用户是否将本篇文章同步到灰金官网。用户确认后，按以下顺序执行：

1. 完整读取 `$PROJECT_ROOT/docs/官网修改与发布安全规则.md` 和 `$PROJECT_ROOT/website/README.md`；
2. 执行 `git status --short --branch`，保留用户和其他 IDE 已有改动，只处理本篇文章涉及的官网文件；
3. 确认标准 Markdown 已在 `$PROJECT_ROOT/content-system/content/articles/`，并核对最终标题、发布日期、摘要、栏目、标签和正文；
4. 先做单篇 dry-run，不得直接全量同步：

```bash
node "$PROJECT_ROOT/website/scripts/sync-column-from-articles.js" \
  --only "{文章源文件名}.md" \
  --dry-run
```

5. 向用户报告预计新增或更新的文章、slug、正文文件和专栏总数；用户确认后，才执行同一条命令并移除 `--dry-run`；
6. 如需使用本篇公众号封面，复制受确认的发布候选图到 `$PROJECT_ROOT/website/public/images/`，使用清晰的英文文件名，并在 `website/public/js/image-urls-v2.js` 登记独立 `imageKey`；没有合适封面时，可复用网站现有专栏图，不得临时使用未确认草图；
7. 运行 `$PROJECT_ROOT/website/scripts/build-column-content-data.js`，确保网页正文内联数据与正文文件一致；
8. 至少验证专栏列表、文章详情、封面图片、标签链接、诊断入口，以及手机/桌面基本显示；
9. 检查 `website/public/js/column-data.js` 中没有重复标题或重复 slug，`website/public/column-content/` 中正文存在且格式沿用网站现有 `.article-body` 结构；
10. 向用户报告本地源代码已同步，但不得把“已写入官网源代码”表述成“生产网站已上线”。只有用户明确说“可以上线”“可以推送”或“发布吧”后，才能另行执行生产发布流程。

如果 dry-run 显示会改动本篇以外的既有文章，必须停止写入，先修正筛选范围；不得用全量覆盖来完成单篇发布。
