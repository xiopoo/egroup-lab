# 灰金官网设计系统与 UI 规范

> 文档性质：官网设计系统真源 / UI 复用规范 / AI 与人工协作约束  
> 适用范围：`website/public/` 下的灰金重组正式官网  
> 当前版本：V0.1  
> 建立时间：2026-07  
> 维护原则：先固定可复用规则，再逐步整理 CSS，不在业务迭代中突然大拆样式结构

---

## 1. 设计定位

灰金官网不是互联网获客落地页，也不是视觉炫技展示页。它的核心任务是建立中小民营企业困境重组领域的可信感、判断力和行动入口。

页面应该给人的第一感受是：

- 克制；
- 专业；
- 可信；
- 有判断顺序；
- 有真实服务经验；
- 能让企业主愿意继续看下去，而不是被营销感推开。

官网设计服务于三个目标：

1. **解释灰金是谁**：灰金不是单一律师、财税或贷款中介，而是围绕经营、财务、债务、资产和法律风险进行综合判断的重组服务团队。
2. **降低初次沟通门槛**：用户可以先通过诊断、服务页、案例和专栏理解问题，再预约沟通。
3. **沉淀专业资产**：页面风格、内容结构和交互都要能长期复用，支持后续案例、文章、诊断工具和服务页持续扩展。

## 2. 核心设计原则

### 2.1 信息优先，装饰靠后

每个页面先回答“用户为什么要看这里”，再考虑视觉表现。装饰元素只能辅助层次，不能抢过标题、正文、按钮和关键事实。

### 2.2 克制比热闹更重要

灰金的行业属性决定了网站不应显得轻佻、浮夸或像模板站。避免过多渐变、图标、标签、阴影、动效和强色块堆叠。

### 2.3 红色只做重点

品牌红 `#AB1942` 是重点色，不是背景填充色。它主要用于：

- 主按钮；
- 激活状态；
- 小标签；
- 关键数字；
- 分隔线或局部强调；
- CTA 区域。

不要把大面积内容区都刷成红色。大面积红色只允许用于品牌 Hero、底部 CTA 或极少数明确的品牌区块。

### 2.4 留白代表专业

页面需要让信息可以呼吸。卡片内部不要塞满标签；列表不要每项都加厚重背景；正文行距应舒展。

### 2.5 专业页面不要做成营销海报

服务、案例、专家、专栏、诊断页面应像专业服务机构网站，不要像课程页、招商页或短视频广告页。

### 2.6 用户路径清晰

重要入口只保留少数几个：

- 开始企业诊断；
- 查看服务方案；
- 查看完整案例或文章；
- 查看完整履历；
- 预约困境咨询 / 预约沟通。

按钮文案必须清楚，不用花哨词。

#### 3.0.5 Logo 使用规则

Logo 使用必须遵守：

- 不拉伸；
- 不压扁；
- 不改比例；
- 不自行换颜色；
- 不加阴影、描边、发光；
- 不裁切文字；
- 不把 Logo 放进复杂纹理或低对比背景；
- 不用截图替代正式素材；
- 不混用多个不同版本造成品牌不一致。

导航栏 Logo 建议保持小而清晰，不要抢过页面内容。页脚 Logo 可以更轻，作为品牌识别即可。

#### 3.0.6 尺寸建议

导航栏横版 Logo 建议：

- 桌面端宽度：`88px` 到 `108px`；
- 移动端宽度：`76px` 到 `88px`；
- 页脚宽度：`48px` 到 `72px`；
- 重要品牌展示区可放大，但必须保证周围留白，不要压迫正文。

Logo 周围至少保留相当于标志高度四分之一的安全空间。不得把 Logo 塞进拥挤按钮、窄标签或复杂纹理背景。

用途：

## 4. 排版系统

### 4.1 字体

官网使用系统字体，不引入外部字体：

```css
-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, "Helvetica Neue", Arial,
"PingFang SC", "Hiragino Sans GB", "Microsoft YaHei", sans-serif
```

原因：

- 中文显示稳定；
- 加载快；
- 不依赖第三方资源；
- 与静态官网定位一致。

### 4.2 标题层级

常用层级：

- 页面 Hero 标题：约 `40px-42px`；
- Section 标题：约 `32px-34px`；
- 卡片标题：约 `20px-24px`；
- 正文：`14px-16px`；
- 说明文字：`12px-14px`。

标题不要使用负字距。官网当前更适合自然字距和稳重行高。

### 4.3 英文小标签

`.section-label` / `.eyebrow` 用于建立章节节奏：

- 字号小；
- 大写；
- 字距略大；
- 使用品牌红；
- 不应变成页面主视觉。

### 4.4 正文

正文行高建议：

- 普通正文：`1.8-1.95`；
- 卡片摘要：`1.7-1.9`；
- 小说明：`1.6-1.75`。

专业页面里，正文可读性优先于视觉紧凑。

## 5. 布局系统

### 5.1 页面宽度

主容器宽度保持在 `1168px-1200px`。不要再回到早期移动端窄条布局。

### 5.2 区块节奏

常规页面：

- 桌面端 section 上下间距：约 `72px-88px`；
- 移动端 section 上下间距：约 `54px-64px`；
- 卡片间距：约 `20px-28px`。

不要为了塞信息而压缩区块间距。

### 5.3 响应式断点

优先维护三类：

- 桌面端：`>=1024px`；
- 平板/小桌面：`768px-1023px`；
- 移动端：`<768px`。

移动端优先保证：

- 单列；
- 按钮可点击；
- 文本不溢出；
- 导航可关闭；
- 弹窗可滚动且不穿透底层页面。

## 6. 组件规范

### 6.1 顶部导航

导航应保持克制：

- 桌面端完整横向链接；
- 移动端汉堡菜单；
- “企业诊断”作为主要行动入口；
- 不在导航上堆太多徽章和解释文字。

### 6.2 内页 Hero

当前稳定模式：

- 品牌红或深色背景；
- 左对齐标题与说明；
- 低透明度 `HUIJIN` 字样作为背景识别；
- 不使用复杂插画；
- 不放卡片式 Hero 文案。

适用页面：

- 关于灰金；
- 品牌主张；
- 专业团队；
- 服务实践；
- 服务方案；
- 案例；
- 专栏；
- 协议和 404 可做简化版本。

### 6.3 Section Header

标准结构：

```html
<div class="section-header">
  <span class="section-label">ENGLISH LABEL</span>
  <h2 class="section-title">中文标题</h2>
  <p class="section-desc">说明文字</p>
</div>
```

使用原则：

- 只解释本区块，不写营销口号；
- 标题不要太长；
- 说明文字不要超过两行到三行；
- 首页可居中，专业内页可左对齐或卡片化。

### 6.4 卡片

卡片用于承载一组独立信息，而不是装饰。

推荐样式：

- 白底；
- 细边框；
- 小圆角 `2px-4px`；
- 轻微 hover 阴影；
- 不使用大圆角、重阴影、花哨渐变。

适用：

- 服务卡片；
- 案例卡片；
- 专家卡片；
- 文章卡片；
- 方法步骤；
- 诊断选项。

### 6.5 专家卡片

当前方向：

- 不使用大姓氏头像块；
- 以姓名、职务、简介和“查看完整履历”为主；
- 编号可以很淡地作为辅助识别；
- “查看完整履历”颜色要轻，不要抢过姓名和职务；
- 弹窗中可以保留小型姓氏标识，用于履历身份识别。

不要做：

- 每人堆 3 个以上标签；
- 使用 AI 假头像；
- 使用图库人物照片冒充团队照；
- 把专家卡片做成营销讲师卡。

### 6.6 案例卡片

案例卡片应突出：

- 行业或问题类型；
- 核心矛盾；
- 处理动作；
- 结果或阶段性成果；
- 脱敏边界。

不要把案例写成“成功学宣传”。涉及客户真实数据时必须脱敏。

### 6.7 服务卡片

服务卡片应表达：

- 服务对象；
- 解决什么问题；
- 交付什么；
- 适合什么阶段；
- 下一步入口。

服务页不要过度强调价格和承诺，避免超出当前能力边界。

### 6.8 CTA

CTA 分为两级：

- 主 CTA：开始企业诊断；
- 次 CTA：预约困境咨询 / 预约沟通。

主 CTA 可用品牌红或白底强按钮。次 CTA 可用透明描边或浅底按钮。不要让两个按钮都同样重。

### 6.9 微信二维码弹窗

微信弹窗是当前咨询入口的关键组件：

- 必须可关闭；
- 移动端二维码图片必须支持长按；
- 弹窗不应遮挡后无法返回；
- 文案保持简短。

### 6.10 弹窗滚动

履历、二维码等弹窗必须：

- 打开时锁住底层页面；
- 弹窗内容滚到底后不带动底层页面；
- 关闭后恢复原页面位置；
- `Escape` 可关闭履历弹窗。

## 7. 页面模板规范

### 7.1 首页

目标：建立第一印象和行动入口。

结构建议：

1. Hero：一句明确主张 + 两个入口；
2. 关键数据或信号；
3. 服务入口；
4. 案例/实践；
5. 专栏或洞察；
6. 联系/诊断 CTA。

首页可以比内页更有视觉冲击，但仍要克制。

### 7.2 服务页

目标：解释服务边界和适用阶段。

重点：

- 服务分类清楚；
- 交付物明确；
- 适合企业类型；
- 诊断和预约入口明显。

不要写过度承诺，不承诺“包解决债务”。

### 7.3 案例页

目标：证明灰金的判断和处理经验。

重点：

- 问题真实；
- 过程可信；
- 数据脱敏；
- 不夸大结果。

### 7.4 专业团队页

目标：建立团队可信感。

重点：

- 团队协同，而不是个人包装；
- 卡片保持疏朗；
- 履历弹窗可读；
- 不用虚假头像；
- 姓名和 title 建议左对齐，形成稳定阅读线。

### 7.5 专栏页

目标：沉淀专业观点和教育市场。

重点：

- 卡片图文清晰；
- 分类和筛选不复杂；
- 文章详情页阅读舒适；
- 法律/财务建议保持免责声明。

### 7.6 诊断页

目标：让用户完成一次低门槛自测。

重点：

- 步骤清晰；
- 问题不吓人；
- 结果可解释；
- 不替代人工诊断；
- 保留后续沟通入口。

### 7.7 协议与 404

目标：可读、可信、不中断用户路径。

协议页应偏文档阅读体验。404 页面可以承担轻量品牌说明，但不要做成夸张错误页。

## 8. 禁止事项

后续官网迭代中，默认禁止：

- 大量使用渐变球、光斑、装饰 orb；
- 大圆角胶囊式 UI 到处堆；
- 把所有按钮都做成同等强度；
- 标签过多导致信息拥挤；
- 使用 AI 生成真人头像冒充团队照片；
- 使用和灰金业务无关的库存人物图；
- 在专业页面使用过度营销语气；
- 对诊断、法律、财务、重组结果做保证性承诺；
- 修改诊断评分、收费、服务承诺等业务规则而不更新战略总纲；
- 在 `website/public` 外另建一套正式官网源码。

## 9. 当前样式资产位置

当前官网样式仍处于“静态网站逐步演进”阶段，不建议立刻大规模拆分 CSS。现有结构先保持：

```text
website/public/common.css
website/public/commercial.css
website/public/site-paper-theme.css
```

### 9.1 `common.css`

定位：基础底座。

包含：

- reset；
- 全局变量；
- container；
- 导航；
- 通用按钮；
- footer；
- 微信弹窗；
- 基础卡片；
- 基础响应式规则。

## 10. 后续 CSS 整理方向

等官网视觉稳定后，可以逐步整理为：

```text
website/public/css/
├── tokens.css
├── base.css
├── components.css
├── pages.css
└── legacy-overrides.css
```

迁移顺序建议：

1. 先抽 `tokens.css`：颜色、字体、间距、圆角、阴影；
2. 再抽 `base.css`：reset、body、container、基础排版；
3. 再抽 `components.css`：nav、hero、card、cta、footer、popover；
4. 最后整理 `pages.css`：各页面局部样式；
5. 旧覆盖先放 `legacy-overrides.css`，逐步删除。

在完成拆分前，不要为了“看起来干净”重写整个网站。


```bash
cd website/public
python3 -m http.server 3001
```

然后检查关键页面返回 200。

视觉检查重点：

- 桌面和手机都不横向溢出；

- 按钮主次清楚；
- 卡片不拥挤；
- 弹窗不滚动穿透；
- 微信二维码可长按；
- 图片不缺失；
- 正文可读。

## 12. AI 协作提示

AI 修改官网时应遵守：

1. 先读 `HJ_战略与开发总纲.md`、`README.md`、`website/README.md` 和本文档。
2. 正式官网只维护 `website/public/`。
3. 优先复用现有类名和组件，不随意发明新风格。
4. 小改优先，避免整站重写。
5. 涉及业务承诺、价格、诊断口径、法律建议时必须让用户确认。
6. 修改后说明完成了什么、用户如何验证、验证结果、尚存风险和建议下一步。
