## 0. 如何使用本文件

本文件是 HJ 仓库的最高级长期说明文件。以后任何 AI、开发者或合作方进入项目，必须先阅读本文件，再阅读对应模块的 README 和 Skill。

它同时承担四个作用：

1. **公司战略总纲**：明确灰金为什么做、服务谁、靠什么形成长期优势。
2. **产品总纲**：明确现有网站、诊断系统、内容系统和未来平台之间的关系。
3. **开发总纲**：明确 Codex 应按什么顺序建设、如何验收、什么不能擅自决定。
4. **协作合同**：明确用户不负责编程，Codex 主导技术实现；用户负责专业判断、真实案例、商业推进和最终业务确认。

本文件不是一次性商业计划，也不是融资宣传稿。所有阶段、价格、收入和时间目标均为待验证假设。只有真实客户行为、有效合同、回款和合规审查才能把假设变成结论。

### 0.1 后续开发的默认阅读顺序

1. 本文件 `HJ_战略与开发总纲.md`
2. 根目录 `README.md`
3. 官网任务：`website/README.md`、`website/package.json`、`website/public/`
4. 内容任务：`content-system/README.md`、`content-system/准备工具.md`
5. 内容工作流：根目录 `.trae/skills/` 下与任务相关的 `SKILL.md`
6. 历史设计依据：根目录 `.trae/specs/` 中与任务相关的规格

### 0.2 文档更新规则

- 产品定位、收费方式、评分口径、法律责任、客户数据用途发生变化时，必须更新本文件。
- 每完成一个阶段，只更新“当前状态”和“验收记录”，不随意重写长期原则。
- 未验证的数据必须标注“目标”或“假设”，不能写成已经实现。
- 代码实际状态与文档冲突时，Codex 应先核验代码，再修正文档，禁止凭旧说明推测。

### 1.2 已有技术栈

- 官网是 **纯静态 HTML/CSS/JavaScript**，不是 React、Next.js 或完整 SaaS。
- 使用 Vercel 托管，根配置的正式输出目录为 `website/public`。

- 诊断问题、权重、计算和报告建议主要写在单个 HTML 文件中。
- 当前统计数据保存在浏览器 `localStorage`，没有真实统一数据库。
- 公众号输出以 Markdown 内容资产、Skill 工作流和微信兼容内联 CSS HTML 为主。

### 1.3 已实现能力

官网已有约 15 个静态页面，包括：

- 首页、品牌介绍、服务介绍、团队/专家；

- 新闻/专栏列表与详情；

内容系统已有：

- 案例检索、术语、诊断式写作、标题、AI 痕迹检测、法律审查、微信排版、发布、保存、恢复和报告等 Skill；
- 多份公众号 HTML 和 Markdown 成品；

本次已增加巴芒复利书房资料域：保留来源隔离，导入股东信、合伙人信、演讲、访谈、问答、概念、公司、人物和书籍等核心 Markdown 资料，并在根 `.trae/skills` 增加通用 `dbs-*` 内容工程 Skill。巴芒资料用于研究、学习和选题，不自动成为灰金原创或客户案例。

### 1.4 当前能力边界

现有系统可以展示、传播和在单台浏览器内完成自测，但还不是可商业化运营的平台：

- 没有用户注册、企业账户、角色和权限体系；
- 没有服务端数据库，无法形成统一企业档案；
- 没有可靠的线索收集、客户授权和后台跟进；
- 没有支付、订阅、白标租户、API 或佣金结算；
- 没有正式的数据脱敏、留存、导出、删除和审计机制；
- 诊断模型尚未形成可审计的版本体系和真实效果验证闭环；
- 网站文章与 `content-system` 尚未自动同步；
- AI 尚未接入文件解析、证据引用和人工复核工作流。

因此，现阶段正确定位是：**成熟的品牌官网与内容生产资产，加上可用的诊断前端原型**。不得对外宣称已经具备完整 SaaS、AI 顾问、行业数据平台或自动决策能力。

## 2. 战略定义

### 2.1 愿景

成为中国中小民营企业困境重组领域，最值得信赖的经营判断与行动支持平台。

### 2.2 使命

把灰金在企业困境识别、债务重组、资产盘活、经营修复和法律风险应对中的真实判断，沉淀为可解释、可复核、可持续改进的知识与工具，帮助企业更早看清问题，更稳妥地采取行动。

### 2.3 唯一北极星

**让更多目标企业通过灰金完成一次可信诊断，并进入可追踪的改善或重组闭环。**

北极星不是页面访问量、文章数量、测评次数或 AI 调用量。核心衡量值是：

> 有效建档企业中，完成诊断并在约定周期内采取至少一项可验证行动的企业比例。

在尚未具备后台前，临时代理指标为：有效诊断完成数、有效咨询线索数和线索转化率。

### 2.4 产品定位

对外产品名称可以继续使用“灰金企业经营健康体检”，因为客户容易理解；对内长期建设对象定义为：

> **灰金企业经营判断平台 = 判断力资产 + 知识资产 + 经授权的数据资产 + AI 辅助能力 + 软件交付能力。**

体检系统是统一入口，不是全部终局。AI 是放大器，不是责任主体。软件是交付载体，不是护城河本身。

### 2.5 战略取舍

未来 24 个月坚持：

- 只服务中小民营企业困境识别与重组相关场景；
- 先解决内部提效和高意向客户转化，再做外部授权；
- 先建立真实案例和结果反馈，再谈行业指数；
- 先做辅助判断，暂不做无人审核的自动法律、财务或重组决策；
- 先验证单区域、单客群、单场景，不同时铺开 C 端、白标、API、政府项目和撮合平台。

明确不做：

- 不为了“像平台”过早建设庞大的多租户和撮合系统；
- 不出售未经验证的评分权威性；
- 不把公众号批量生产等同于专业内容资产；
- 不使用未经授权的客户数据训练或展示；
- 不承诺“自动生成完整重组方案”“替代律师/会计师/顾问”。

## 3. 核心客户与价值主张

### 3.1 第一优先客户：已出现经营或债务信号的中小民企负责人

典型问题：看不清严重程度、不知道先处理什么、信息分散、害怕错误行动扩大风险。

灰金交付：低门槛初筛、结构化深度诊断、风险优先级、90 天行动建议、必要时转入人工专项服务。

### 3.2 第二优先客户：现有咨询与重组服务客户

典型问题：重复收集材料、报告制作耗时、项目过程不透明、复盘数据缺失。

灰金交付：统一企业档案、材料清单、版本化诊断、节点跟踪、标准报告和人工复核记录。

### 3.3 第三优先客户：律所、财税和企服渠道

只有在内部闭环稳定后验证。价值不是简单“贴牌问卷”，而是帮助渠道筛选客户、识别风险、规范转介并提高成交质量。

### 3.4 暂缓客户

银行、AMC、政府、园区、征信机构、跨区域生态平台属于后续潜在客户。在数据规模、合规能力、模型验证和服务能力达到门槛前，不作为第一阶段开发依据。

## 4. 复利资产模型与护城河

### 4.1 判断力资产（第一优先）

包括问题体系、风险信号、判断规则、适用条件、反例、方案边界和人工复核理由。每条判断必须能回答：来源是什么、适用于谁、什么时候失效、谁审核过。

### 4.2 知识资产

包括真实文章、概念、术语、案例、法律依据、模板、复盘和常见问题。现有 `content-system` 是知识生产起点，不只是营销工具。

### 4.3 数据资产

只有满足合法授权、目的限定、最小收集、安全保存、可删除和可追溯的数据，才可称为数据资产。测评次数本身不是护城河；带有输入质量、人工判断、采取行动和最终结果的闭环样本才有价值。

### 4.4 AI 与软件资产

AI 用于材料抽取、知识检索、报告初稿、风险提示和质检；软件用于规模化交付。二者必须建立在人审、证据引用、版本记录和权限控制上。

### 4.5 品牌与网络资产

专业内容、真实结果和可信服务形成品牌；渠道合作和服务网络只有在标准、质量控制、投诉处理和利益分配成熟后才形成网络效应。

### 4.6 核心飞轮

```text
专业内容与渠道触达
        ↓
目标企业完成初筛
        ↓
人工确认的有效诊断与企业档案
        ↓
企业采取行动并记录结果
        ↓
案例、规则和模型复盘
        ↓
更可信的诊断、内容与服务
        ↓
更高质量的客户与合作方
        ↺
```

飞轮的关键不是“更多数据”，而是“更多经授权、经复核、有结果反馈的高质量样本”。

## 5. 产品架构

### 5.1 统一底座

未来平台由六个层次组成：

1. **品牌与内容入口**：现有官网、公众号内容、案例和服务介绍。
2. **诊断入口**：免费快速筛查、深度诊断、报告和咨询转化。
3. **企业档案**：企业资料、授权、历史测评、文档、服务记录和行动结果。
4. **判断与知识引擎**：版本化评分规则、知识检索、案例依据和人工复核。
5. **服务工作台**：线索分级、诊断报告、任务、随访、项目节点和结果复盘。
6. **商业化接口**：在验证后增加订阅、渠道、API、数据产品与生态合作。

### 5.2 现有产品的保留与改造

| 现有资产 | 决策 | 下一步 |
|---|---|---|
| `website/public` 静态官网 | 保留 | 统一导航、页面数据源、可访问性和转化追踪 |
| `diag.html` 诊断原型 | 保留业务逻辑，逐步拆分 | 先版本化题库与评分，再接真实后端 |
| 网站案例/新闻静态页 | 保留外观 | 与内容索引建立单一数据来源 |
| `content-system` 内容库 | 作为知识底座起点 | 增加来源、审核状态、关联概念和版本字段 |
| `.trae/skills` 内容流程 | 保留并迁移适配 Codex | 逐项验证触发规则、输入输出和合规检查 |
| 根目录 `public` 测试页 | 不作为正式源 | 确认无用后再由用户批准清理 |

### 5.3 诊断产品分级

**免费快速筛查**：少量核心问题、风险等级、有限建议、明确免责声明，引导用户授权留下联系方式。

**专业深度诊断**：更完整问卷、材料补充、五维分析、证据说明、人工复核和可下载报告。定价在真实访谈和付费测试后确定。

**专项人工服务**：专项诊断、年度陪跑和全案服务。系统负责标准化和留痕，专业人员对最终结论负责。

**渠道版/API 版**：属于后续产品，不在第一阶段承诺开发日期或固定价格。

## 6. AI 平台原则

### 6.1 AI 可以做什么

- 从用户授权材料中提取字段，并标记来源页；
- 根据已审核知识库检索相关规则、案例和风险提示；
- 生成报告初稿、材料缺口和复核清单；
- 检查内容中的 AI 痕迹、术语、来源、法律风险和品牌格式；
- 帮助顾问复盘案例，提出待确认问题。

### 6.2 AI 不能独立做什么

- 不能给出无证据来源的确定性法律或财务结论；
- 不能代表灰金承诺重组成功、债务减免或司法结果；
- 不能在无人审核时直接向客户发送高风险正式方案；
- 不能擅自使用客户数据训练模型、制作案例或行业报告；
- 不能用自动化评分代替完整尽调。

### 6.3 每个 AI 输出的最低要求

- 显示资料来源或明确说明“依据用户自述”；
- 区分事实、推断、建议和待确认事项；
- 记录使用的知识库与规则版本；
- 对高风险结论设置人工审核；
- 保存修改和批准记录；
- 对外报告包含适用范围与免责声明。

## 7. 数据、模型与合规

### 7.1 数据分类

1. 公开内容：官网、公开文章、公开法规和公开案例。
2. 运营数据：访问、内容表现、转化和服务状态。
3. 企业一般数据：行业、规模、经营指标和诊断答案。
4. 企业敏感数据：财务报表、流水、合同、债权债务、诉讼和个人担保。
5. 个人信息：姓名、电话、微信、身份信息和联系方式。

不同类型必须采用不同权限、保存周期和导出规则。

### 7.2 上线前最低合规门槛

- 明确隐私政策、用户协议、诊断免责声明和数据授权文本；
- 做到最小必要收集，并说明使用目的；
- 敏感字段传输和存储加密；
- 企业、渠道、员工之间严格权限隔离；
- 支持数据查询、更正、导出和删除流程；
- 记录登录、查看、修改、导出和 AI 使用行为；
- 正式对外前由合格的中国法律专业人士审核。

### 7.3 评分模型治理

当前五维模型和权重是业务原型，不应直接宣称具有统计学预测能力。正式商业化前必须建立：

- 题目编号、版本、维度、分值、权重、适用范围和制定依据；
- 规则变更记录以及新旧报告可比较方式；
- 数据缺失、矛盾答案和异常值处理；
- 专家复核与覆盖自动评分的流程；
- 预测目标，例如“未来 90 天现金流风险”或“是否需要人工介入”；
- 样本外验证、误报/漏报分析和定期校准。

“500 份样本”“1000 份样本”只能作为运营目标，不能自动证明模型有效。有效性取决于样本代表性、标签质量和结果反馈。

## 8. 分阶段开发路线图

时间是参考，不是硬承诺。每阶段必须满足退出条件，才能进入下一阶段。

### 阶段 0：仓库稳定与事实统一（进行中）

**目标**：确认唯一正式代码源，避免在混乱目录上继续叠加功能。

Codex 主导：

- 完成目录和 Git 变更审计；（已完成）
- 确认 `website/public` 是正式网站源；（已完成）
- 恢复根 `.trae`，建立 Trae/Codex 双 IDE 共用工作区；（已完成）
- 为页面、诊断、内容和 Skill 建立资产清单；
- 建立最小检查：链接、图片、主要页面打开、移动端基本显示；
- 将现有业务状态和代码状态的矛盾列入待办。

用户负责：

- 确认官网域名、正式联系方式、服务名称和当前价格；
- 确认哪些历史文件可以清理；
- 指定对外可公开与不可公开内容。

**退出条件**：生产源目录唯一；本地可预览；主要页面无阻断错误；用户确认资产清单。

### 阶段 1：现有漏斗跑通（预计 2—6 周）

**目标**：把“内容/官网 → 诊断 → 留资 → 人工跟进”做成可测量闭环。

优先功能：

- 诊断题库与评分规则从页面代码中分离并版本化；
- 新增明确授权的线索表单与服务端保存；
- 记录来源、诊断完成、咨询点击和跟进状态；
- 建立简单内部线索工作台或可靠导出；
- 统一公众号内容到网站的发布流程；
- 补齐隐私、协议、免责声明与错误处理。

**退出条件**：至少连续 4 周稳定收集真实漏斗数据；每条线索可追溯来源和跟进结果；敏感数据不再只依赖浏览器本地存储。

### 阶段 2：内部企业档案与顾问提效（预计 2—6 个月）

**目标**：先让系统成为灰金内部每天真正使用的交付工具。

优先功能：

- 企业唯一档案、联系人、授权和材料清单；
- 历史诊断、评分版本和人工复核；
- 报告初稿、审阅、批准、导出和交付记录；
- 随访任务、行动计划和结果回填；
- 角色权限、审计日志、备份与恢复；
- 内容/案例/概念和企业复盘之间的受控关联。

**退出条件**：新增服务客户原则上全部建档；核心顾问连续使用；重复录入显著下降；报告均有版本和审批记录；完成一次恢复演练。

### 阶段 3：AI 辅助顾问（预计 6—12 个月）

**目标**：用 AI 缩短材料整理和报告初稿时间，不降低专业责任标准。

优先功能：

- 上传文档的文字提取、字段识别和来源定位；
- 受审核知识库检索；
- 报告初稿与缺失材料清单；
- 事实/推断/建议标记；
- 高风险结论强制人审；
- 质量评测集、错误记录和成本监控。

**退出条件**：在内部评测集达到预设准确率；重大错误可追踪；顾问实际节省时间；所有对外输出经过责任人批准。

### 阶段 4：专业版商业化验证（预计 9—18 个月）

**目标**：验证客户是否愿意为持续诊断和行动支持付费。

开发内容按实验决定，可能包括账户、订阅、复测、报告对比和服务权益。先用小批量人工辅助方式验证价格与留存，不先建设完整复杂订阅系统。

**退出条件**：形成真实付费、续费、使用频率和服务转化数据；明确最适合的客户画像；单位经济模型可解释。

### 阶段 5：渠道与数据产品（18 个月以后，达标才启动）

候选方向：渠道工作台、有限白标、标准接口、匿名聚合报告。启动门槛包括内部系统稳定、客户授权明确、数据质量达标、渠道愿意付费、支持成本可控。

服务商撮合、资产撮合、行业评级和跨区域复制必须单独立项，重新做法律、利益冲突、资质、质量和声誉风险评估，不能视为诊断系统的自然延伸。

## 9. 商业模式与验证顺序

### 9.1 收入优先级

1. 现有高价值人工服务：提供现金流与真实案例。
2. 专业诊断与年度复测：验证产品化价值。
3. 渠道工具：验证 B 端筛选和转介价值。
4. AI 辅助与数据接口：在准确性、合规和支持能力成熟后收费。
5. 数据报告和生态分润：长期选项，不提前计入确定收入。

### 9.2 定价原则

原 V1.0 中的 499 元/年、3980 元/年、API 5 万元起以及服务价格可作为访谈锚点，不作为已确定产品价格。最终定价至少依据：

- 目标客户愿付价格；
- 获客、人工审核、模型调用、客服和合规成本；
- 使用频率、续费率和服务转化；
- 渠道是否产生额外实施与支持成本；
- 错误判断可能带来的责任成本。

### 9.3 “被动收入”的现实定义

系统收入仍需要开发、运维、安全、客服、合规、模型校准和品牌维护，不使用“纯被动、无人值守、边际成本为零”作为内部决策前提。更准确的目标是：

> 单位新增收入所需的边际人工时间持续下降，同时质量、合规和客户结果不恶化。

## 10. 指标体系

### 10.1 第一阶段漏斗指标

- 目标访客到诊断开始率；
- 诊断完成率；
- 同意留资率；
- 有效线索率；
- 首次响应时间；
- 咨询预约率与服务成交率；
- 不同内容、渠道和页面带来的有效客户质量。

### 10.2 交付与客户结果指标

- 建档完整率、材料补齐周期；
- 报告制作工时和人工修改率；
- 诊断后 30/90 天行动完成率；
- 风险变化、现金流改善或关键节点完成情况；
- 客户满意度、投诉、退款和重大错误。

### 10.3 模型与 AI 指标

- 字段抽取准确率、引用正确率；
- 误报率、漏报率、高风险错误数；
- 专家覆盖自动判断的比例与原因；
- 单份报告 AI 成本、处理时间和节省工时；
- 不同模型/规则版本的效果对比。

### 10.4 商业指标

- 获客成本、回款周期、毛利率；
- 付费转化、续费、流失和服务升级；
- 每个渠道的收入、支持成本与坏账；
- 软件收入占比可以跟踪，但不作为牺牲客户结果的目标。

## 11. Codex 与用户的职责边界

### 11.1 Codex 主导

- 阅读仓库、制定技术方案和拆分任务；
- 编写、修改、测试和说明代码；
- 维护架构、数据结构、接口、自动检查和部署文档；
- 主动识别安全、隐私、数据和维护风险；
- 用非技术语言汇报“完成了什么、如何验证、用户下一步做什么”；
- 在不改变商业范围的前提下，选择合理技术实现。

### 11.2 用户主导

- 定义真实业务优先级和客户痛点；
- 提供并确认专业判断、案例事实、收费和服务边界；
- 推进客户访谈、试用、销售、合作和回款；
- 决定品牌表达和重大商业取舍；
- 安排律师、会计师等专业人员审核需要资质或责任承担的内容。

### 11.3 必须由用户确认后才能执行

- 对外发布、生产部署、收费和客户通知；
- 收集新的敏感数据或改变数据用途；
- 删除历史数据、大范围迁移或破坏性仓库操作；
- 接入付费服务、签订第三方合同或产生持续费用；
- 改变诊断结论、定价、正式服务范围或法律免责声明；
- 使用真实客户案例、数据或材料进行宣传、研究或模型训练。

### 11.4 用户提交新需求的简单模板

用户不需要写技术说明，只需回答：

```text
我希望谁使用：
他现在遇到什么问题：
我希望他完成什么动作：
什么结果算成功：
是否涉及真实客户数据、收费或对外发布：
```

信息不足但风险较低时，Codex 应做合理假设并继续；涉及收费、数据用途、法律责任或不可逆变更时必须先确认。

## 12. 每个开发任务的强制流程

### 12.1 开始前

1. 阅读本文件和相关模块说明。
2. 检查 Git 状态，识别用户已有改动。
3. 先验证当前功能，不凭文档猜测。

2. 业务规则与页面显示分离，重要规则必须版本化。
3. 不把密钥、客户资料和敏感信息写进代码仓库。
4. 对移动端、错误状态、空数据、慢网络和重复提交做基本处理。

至少验证：

- 主要用户流程可完成；
- 手机和桌面关键页面无明显阻断；
- 链接、图片、表单和错误提示正常；
- 未破坏已有页面和内容工作流；
- 涉及数据时已检查权限、日志、备份和删除路径；
- 更新相关 README、待办和本文件状态（如适用）。

## 13. 内容系统长期规则

### 13.1 内容的三重作用

每份内容应至少服务一个目标：

1. 帮助目标客户理解问题并采取行动；
2. 沉淀灰金可复用的专业判断；
3. 为诊断、报告和顾问服务提供经审核的知识依据。

流量不是唯一目标，不能为高产量牺牲真实性和专业可信度。

### 13.2 标准生产链

```text
选题/真实问题
→ 资料与案例核验
→ 诊断式写作
→ 标题与结构检查
→ AI 痕迹检测
→ 术语和法律检查
→ 微信 HTML 排版
→ 人工批准与发布
→ 保存原文、成品和数据
→ 拆解概念、案例、金句与后续问题
```

现有根目录 `.trae/skills` 是这条链路的执行基础。Trae 与 Codex 使用同一套 Skill；应保持每个 Skill 的单一职责，不把所有流程合成不可审计的一次生成。

### 13.2.1 多领域路由

- 灰金重组内容：使用 `hj-*`，写入灰金文章、概念、金句和公众号成品；
- 巴芒研究内容：使用 `dbs-*`，读取 `content/bamang`，产出研究笔记、学习内容和选题素材；
- 跨域内容：在 `content/content-catalog.json` 中登记来源域，文章中区分原文依据、研究启发和灰金实务判断；
- `gray-gold-imports` 只表示“待审核导入”，不能直接发布。

### 13.3 知识条目最低字段

- 唯一编号和类型；
- 标题、摘要、正文；
- 来源和发布日期；
- 作者/审核人、审核状态和审核时间；
- 适用范围、风险提示和失效条件；
- 关联概念、案例、文章和诊断规则；
- 版本和修改记录；
- 是否允许对外公开、是否允许用于 AI 检索。

## 14. 关键风险与止损条件

### 14.1 战略风险

- 同时建设过多产品导致现金流和注意力分散；
- 把“有系统”误认为“有需求”；
- 把内容数量和样本数量误认为护城河；
- 过早宣传行业标准或权威评分损害信任。

### 14.2 技术风险

- 当前单页代码继续膨胀，难以维护和测试；
- 静态站点承担敏感数据和商业后台职责；
- 浏览器本地统计被误当成真实运营数据；
- 第三方 AI、分析和支付服务带来数据外流或锁定。

### 14.3 业务与法律风险

- 自动建议被客户理解为正式法律/财务结论；
- 使用真实案例、财务或诉讼数据没有充分授权；
- 撮合和评级业务产生资质、收费、利益冲突与声誉风险；
- 渠道服务质量失控反噬灰金品牌。

### 14.4 暂停或调整条件

出现以下任一情况，应暂停扩张并复盘：

- 连续两个验证周期没有有效客户使用或付费；
- 客户投诉或重大错误超过可接受阈值；
- 人工审核成本随规模同步增长，未形成效率提升；
- 数据授权和安全能力无法满足真实业务；
- 新产品明显影响现有高价值服务交付；
- 商业化依赖虚假“被动收入”或未经验证的行业权威叙事。

## 15. 未来 3—5 年的合理终局

如果前述阶段逐一验证成功，灰金将从“依赖创始人个人交付的咨询公司”升级为：

> 以真实专业判断为核心、以经授权的闭环数据为燃料、以 AI 和软件为放大器的企业经营判断与行动支持平台。

成熟状态应表现为：

- 官网、内容、诊断、企业档案和顾问服务使用同一知识与数据底座；
- 标准工作由系统完成，高风险判断由专业人员负责；
- 新客户带来的高质量反馈持续改善知识和服务；
- 收入不再完全绑定创始人时间，但仍保留必要的人审、服务和合规成本；
- 平台是否扩展到渠道、API、指数和撮合，由真实市场验证决定，而不是由愿景倒推。

终局不是做一个功能繁多的 SaaS，也不是追求“无人值守”。终局是让灰金的专业判断可以被稳定传承、反复验证、规模化交付，并持续帮助企业产生更好的实际结果。

## 18. 一句话战略

**灰金不以“做一个体检软件”为终点，而是先用现有官网、诊断原型和内容系统跑通真实客户闭环，再把经过验证的判断、知识和数据沉淀为可信平台；用户负责专业与商业，Codex 负责把它安全、清晰、分阶段地建设出来。**
